# 失落的亚特兰大 Demo 开发规范

## 1. 目的

本项目是 Demo，但后续代码必须按照“可扩展、可替换、低耦合”的方式编写。

这份文档的目标是提前定规则，避免后面加入以下系统时推倒重来：

- 天气系统
- 背包系统
- 商店系统
- 随机事件系统
- NPC / 阵营系统
- 数值 Buff / Debuff 系统

本规范从现在开始生效。后续新增代码默认都应遵守。

## 2. 核心原则

### 2.1 单一职责

每个系统只做一件事。

- `GameManager` 负责全局流程与核心状态
- `SaveManager` 负责存档读写
- UI 脚本负责表现与交互
- 功能系统负责自己的业务逻辑

禁止把“流程控制 + UI 控制 + 数值修改 + 存档处理”全塞进一个脚本。

### 2.2 面向阶段，而不是面向彼此

后续系统不应该彼此硬调用，而应该围绕“游戏阶段”响应。

正确思路：

- 天气系统监听“新的一天开始”
- 背包系统监听“搜索结果结算”
- 商店系统监听“进入交易阶段”

错误思路：

- `WeatherSystem` 直接调用 `InventorySystem`
- `ShopSystem` 直接改 UI 节点
- 任意脚本都能直接改全局状态

### 2.3 数据驱动优先

能放配置的内容，不要写死在逻辑里。

优先配置化的内容：

- 事件文本
- 物品定义
- 商店商品池
- 天气类型与效果
- 数值修正表

## 3. 全局阶段规范

后续所有核心玩法，必须围绕统一的“阶段”推进。

建议全局阶段如下：

1. `boot`
2. `main_menu`
3. `run_start`
4. `day_start`
5. `day_action_select`
6. `day_action_resolve`
7. `night_event`
8. `day_end`
9. `game_over`

说明：

- `boot`：项目启动后的初始化阶段
- `main_menu`：主菜单阶段
- `run_start`：开始新游戏或读档后，正式进入流程前的准备阶段
- `day_start`：新一天开始，结算每日开始效果
- `day_action_select`：玩家选择行动
- `day_action_resolve`：对行动进行结果结算
- `night_event`：夜间强制事件
- `day_end`：一天结束，总结算、自动存档、进入下一天
- `game_over`：死亡或失败结算

## 4. Hook 规范

为了可扩展性，后续系统应优先通过 Hook 接入全局流程。

### 4.1 标准 Hook 时机

每个阶段默认支持以下 4 类 Hook：

1. `before_enter`
2. `after_enter`
3. `before_exit`
4. `after_exit`

举例：

- `day_start.before_enter`
- `day_start.after_enter`
- `night_event.before_exit`

### 4.2 Hook 的使用目的

- `before_enter`：阶段进入前修正上下文
- `after_enter`：阶段进入后执行附加逻辑
- `before_exit`：阶段离开前补充结算
- `after_exit`：阶段离开后做清理或派发后续影响

### 4.3 Hook 设计约束

Hook 应满足以下要求：

- 可以有多个订阅者
- 不要求订阅者之间互相认识
- 支持优先级
- 支持按系统注销
- 不允许依赖场景树中的具体 UI 路径

### 4.4 Hook 命名建议

统一使用：

`阶段名.时机`

例如：

- `day_start.after_enter`
- `day_end.before_exit`
- `night_event.after_enter`

如果后续需要非阶段型 Hook，统一使用业务域前缀：

- `inventory.item_added`
- `weather.changed`
- `shop.trade_completed`

## 5. 模块边界规定

### 5.1 GameManager 的职责边界

`GameManager` 只应该负责：

- 当前天数
- 核心生存资源
- 当前全局阶段
- 开始游戏 / 结束游戏 / 推进天数
- 对外提供统一流程入口

`GameManager` 不应该负责：

- 天气效果细节
- 背包物品规则
- 商店刷新逻辑
- 具体 UI 控件控制
- 单个事件的文本内容

### 5.2 UI 层职责边界

UI 层只负责：

- 展示数据
- 接收玩家输入
- 把输入转换为“意图”

UI 层不负责：

- 直接推进游戏核心流程
- 修改多个系统状态
- 处理存档结构

举例：

- 按钮可以发出“开始新游戏”的请求
- 按钮不应该自己计算资源变化

### 5.3 功能系统职责边界

未来新增功能时，应该拆成独立系统。

例如：

- `WeatherSystem`
- `InventorySystem`
- `ShopSystem`
- `EventSystem`

每个系统只管理自己的数据、规则和存档片段。

## 6. 系统之间的通信规则

后续代码默认遵循以下优先级：

1. 优先通过阶段 Hook 通信
2. 其次通过事件/消息通信
3. 最后才允许有限度的直接调用

### 6.1 允许直接调用的情况

仅在以下情况允许：

- 明确的主控对象调用子对象
- 同一功能域内部的小范围调用
- 无法抽象成阶段或事件的底层工具函数

### 6.2 禁止直接调用的情况

以下情况默认禁止：

- 系统 A 直接修改系统 B 的内部字典
- UI 脚本直接改多个业务系统状态
- 任意脚本直接跳场景并同时修改全局数据
- 为了省事把一个系统做成“全能单例”

## 7. 存档规范

后续存档必须支持按系统拆分。

建议结构：

```json
{
  "meta": {
    "version": 1,
    "save_time": "2026-06-25T00:00:00"
  },
  "run_state": {
    "day": 3,
    "phase": "night_event",
    "resources": {
      "food": 90,
      "morale": 70,
      "stamina": 60
    }
  },
  "systems": {
    "weather": {},
    "inventory": {},
    "shop": {},
    "event": {}
  }
}
```

规定：

- 全局状态放在 `run_state`
- 各系统数据放在 `systems.<system_id>`
- 每个系统只读写自己的存档片段
- 禁止一个系统接管整个存档结构

## 8. 数据与配置规范

后续新增内容时，优先把“内容定义”和“逻辑实现”拆开。

### 8.1 应该配置化的内容

- 事件表
- 物品表
- 商店商品表
- 天气权重表
- 阶段效果表

### 8.2 不应该写死的内容

- 事件文案直接塞进主流程脚本
- 物品效果写进 UI 按钮回调
- 商店价格散落在多个文件

## 9. 文件组织建议

建议后续按职责扩展目录：

```text
scripts/
├── autoload/
│   ├── GameManager.gd
│   ├── SaveManager.gd
│   └── PhaseManager.gd
├── systems/
│   ├── WeatherSystem.gd
│   ├── InventorySystem.gd
│   ├── ShopSystem.gd
│   └── EventSystem.gd
├── ui/

│   ├── HUD/
│   └── Panels/

    ├── shops/
    └── weather/
```

## 10. 未来扩展示例

### 10.1 天气系统

天气系统应在这些阶段参与：

- `day_start.after_enter`：刷新当天天气
- `day_action_resolve.before_enter`：对行动结果添加修正
- `night_event.before_enter`：影响夜间事件权重

天气系统不应该：

- 直接修改背包 UI
- 直接切换场景
- 自己决定是否进入下一阶段

### 10.2 背包系统

背包系统应在这些阶段参与：

- `day_action_resolve.after_enter`：发放拾取物品
- `night_event.after_enter`：消耗或损坏物品
- `day_end.before_exit`：整理每日结算数据

背包系统不应该：

- 接管 `GameManager` 的资源逻辑
- 直接写主菜单状态

### 10.3 商店系统

商店系统应在这些阶段参与：

- 特定交易阶段 `shop_open.after_enter`
- `day_end.after_enter`：刷新商品池

商店系统不应该：

- 自己维护“当前是第几天”的唯一真值
- 绕过统一资源接口直接扣钱

## 11. 代码风格规定

### 11.1 新增功能时必须先想扩展点

每次写功能前，先问自己两个问题：

1. 这个功能以后会不会加分支？
2. 这个功能以后会不会被别的系统影响？

如果答案是“会”，就不要写成硬编码直连。

### 11.2 禁止把 Demo 当一次性脚本写

即使是 Demo，也禁止以下做法：

- 复制粘贴一堆相似逻辑
- 用超长函数串完整个游戏流程
- 让场景脚本承担业务核心
- 把未来可能扩展的配置写死

### 11.3 优先保留接口

即使当前功能还没实现完整，也应该优先保留好接口位置。

例如：

- 预留阶段切换入口
- 预留系统存档接口
- 预留阶段 Hook 注册入口

## 12. 禁止事项

从现在开始，以下写法视为不符合规范：

1. 在 UI 脚本里直接写核心流程结算
2. 在任意脚本里随意切场景并顺手改全局状态
3. 新系统直接依赖另一个系统的内部字段
4. 把多个系统的存档混在一个大字典里无命名空间
5. 用大量 `if/elif` 把所有未来功能堆进 `GameManager`
6. 为了图快，把天气、背包、商店都写成一个脚本

## 13. 后续落地建议

这份规范对应的下一步技术落地，建议顺序如下：

1. 增加 `PhaseManager`
2. 统一阶段切换入口
3. 给系统预留 Hook 注册机制
4. 给存档改成分系统结构
5. 再开始做天气、背包、商店等功能

## 14. 一句话结论

后续所有新增功能，都必须接入“全局阶段 + Hook + 系统解耦 + 分系统存档”的架构，不允许继续沿着临时脚本式写法扩散。
