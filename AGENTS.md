# Codex 项目约束

在本项目中进行任何代码修改前，必须先读取并遵守 `开发规范.md`。

## 强制要求

- 每一次写完代码都要测试一遍，看有没有报错和逻辑漏洞。测试完成再结束工作
- 后续功能必须围绕“全局阶段 + Hook”设计。
- 系统之间保持解耦，不互相硬调用。
- UI 只负责展示和输入，不承载核心业务逻辑。UI相关的功能要做好兼容性，确保多种分辨率下正常
- 存档必须按系统拆分，避免一个系统接管整个存档结构。
- 新增天气、背包、商店、事件、NPC 等功能时，必须预留扩展点。
- 即使当前项目是 Demo，也不能把后续可能扩展的逻辑写成一次性脚本。
- 每次完成和游戏进度有关的改动后，必须更新开发进度文档；开发进度文档只记录当前项目进度、已完成状态、关键决策和下一步方向，不记录中间过程、临时尝试或修改流水。

## 工作流程

1. 开始修改代码前，先阅读 `开发规范.md`。
2. 判断当前改动属于哪个阶段、哪个系统、哪个职责边界。
3. 优先通过阶段 Hook、事件或系统接口通信。
4. 修改完成后，检查是否违反 `开发规范.md` 中的禁止事项。
5. 如果改动影响游戏进度或项目状态，更新开发进度文档，并确保只写阶段性结果，不写过程流水。

## 默认架构方向

本项目后续应逐步落地：

1. `PhaseManager`
2. 统一阶段切换入口
3. Hook 注册与触发机制
4. 分系统存档结构
5. 独立功能系统，例如 `WeatherSystem`、`InventorySystem`、`ShopSystem`、`EventSystem`
